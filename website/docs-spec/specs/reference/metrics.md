# Spec: reference/metrics.mdx
Status: generated (new mechanism; replaces the hand-maintained `reference/metrics.mdx`)
Audiences: A4 (test-deployment operators), A5 (contributors)

Goal: The reader can look up every Prometheus metric FrogDB exports — its name,
type (counter/gauge/histogram), label set, and help text — grouped by category,
and knows where to scrape it. The table is generated from the typed metric
definitions, so it cannot drift from the binary (the current page's own Aside
admits it is hand-maintained and can lag the source — that is the problem this
spec removes).

Not in scope:
- Alerting thresholds, dashboards, PromQL recipes — those belong to
  Operations → Observability / Diagnostics; link, do not duplicate.
- OTLP export / tracing configuration — Operations → Observability.
- Metric *values'* operational interpretation beyond the one-line help text.

Sources of truth (author must read):
- `frogdb-server/crates/telemetry/src/definitions.rs` — the single
  `define_metrics! { … }` invocation is the authoritative list. VERIFIED: 72
  metrics, each declared as `<type> StructName("frogdb_name") { labels: [...] }`
  with a `///` doc comment that becomes help text.
- `frogdb-server/crates/metrics-derive/src/lib.rs` — the proc-macro. VERIFIED it
  already emits, in addition to per-metric structs, a runtime registry:
  `pub static ALL_METRICS: &[MetricDefinition]` and `pub const METRICS_COUNT`.
- `frogdb-server/crates/telemetry/src/typed.rs` — `MetricDefinition { name,
  help, metric_type, labels }` (VERIFIED fields) and `MetricType` enum; plus a
  `MetricDefinition::category()` helper (typed.rs:59) that returns the token
  after the `frogdb_` prefix.
- `frogdb-server/crates/telemetry/src/lib.rs:49` — re-exports `ALL_METRICS`,
  `METRICS_COUNT`.

Existing content:
- Replace: `reference/metrics.mdx`. Its category groupings and help strings are a
  useful reference for the *desired* human-facing categories, but the data must
  now come from `ALL_METRICS`. Delete its self-flagged "manually maintained"
  Aside. Fix its `See Also` links (`operations/monitoring` and
  `operations/performance` are being merged away per PLAN §4 — point at
  Observability / Diagnostics).

Extraction mechanism — recommendation (work item S3, §6):
Use the **existing runtime registry**, not text parsing. `ALL_METRICS` is
already generated by the derive macro and gives `{name, help, metric_type,
labels}` for all 72 metrics with zero new proc-macro work. Recommended pipeline:
- Add a tiny generator that iterates `frogdb_telemetry::ALL_METRICS` and writes
  `website/src/data/metrics.json` — an array of
  `{ name, type, labels: [...], help, category }`. Implement it either as:
  - (a, recommended) a `--metrics`/subcommand mode of the existing `docs-gen`
    binary (it already links the workspace and owns the `--check` convention), or
  - (b) a `#[test]`-driven emitter in the telemetry crate that writes the JSON.
  Prefer (a) for one consistent generation entry point and `--check` flow.
- This is introspection of a `const`/`static` — no runtime server needed, no
  `--help`-style capture, no parsing of `definitions.rs` source. That makes it
  the most robust of the three options (runtime introspection vs source parsing
  vs registry function): the registry function already exists.

Category handling — VERIFY-BEFORE-WRITING:
`MetricDefinition::category()` naively takes the first `_`-delimited token after
`frogdb_`, which does NOT reproduce the current page's human groupings. Concrete
mismatches: the three version/upgrade metrics split into `active` / `binary` /
`version`; scatter-gather → `scatter`; split-brain → `split`; WAL + snapshot both
intended under "Persistence". So `category()` alone will not produce the page's
sections. Decide one of:
  - Add an explicit `category` (or section) to each metric in `define_metrics!`
    (extend the macro to accept and emit it) — most robust, single source.
  - Or maintain a small prefix→category map in the generator (e.g. `active`,
    `binary`, `version` → "Version / Rolling Upgrade"; `wal`, `snapshot`,
    `persistence` → "Persistence"). Simpler, but a second place to edit.
Record the choice; the page structure below assumes categorized output exists.

Structure (H2/H3 outline):
- Intro (2 sentences): FrogDB exports Prometheus metrics at `/metrics` on the
  HTTP endpoint (default port from `<ConfigDefault path="http.port" />`);
  enable via config. Link to Operations → Observability for scraping/OTLP.
- One H2 per category (System, Connections, Commands, Keyspace, Shards,
  Persistence, Pub/Sub, Network, Memory/Eviction, Blocking, Lua, Transactions,
  Scatter-Gather, Split-Brain, Latency, Version/Rolling Upgrade — final set
  driven by the chosen category source). Each contains a generated table:
  columns Metric, Type, Labels, Description — rendered by a new
  `MetricsTable.astro` component filtering `metrics.json` by category (mirrors
  `ConfigTable.astro`).
- H2 "See also": Operations → Observability, Operations → Diagnostics.

Generated data / component (work item S3, §6):
- `website/src/data/metrics.json` from `ALL_METRICS` (see mechanism above).
- New `MetricsTable.astro` component (model on `ConfigTable.astro`) taking a
  `category` prop and throwing at build time on an unknown category — same
  section-coverage safety as ConfigTable.
- Justfile: add generation as a `docs-build` / `docs-dev` prerequisite alongside
  `docs-gen`; give it a `--check` recipe (`metrics-gen-check`).

Drift guards:
- Data generated from `ALL_METRICS`; adding/removing a metric in
  `definitions.rs` regenerates the JSON. A `metrics-gen --check` CI job
  (analogous to `docs-gen-check`) fails the build when `metrics.json` is stale,
  and must also trigger on `website/**` and telemetry-crate changes (S8).
- Optional stronger guard: assert `metrics.json` length equals `METRICS_COUNT`
  so a metric can never be dropped from the page silently.
- Category-coverage: the `MetricsTable` build-time throw catches a page H2 whose
  category has no metrics; add a check that every `metrics.json` category maps to
  exactly one H2 to catch the reverse (a new category with no section).
- No hand-typed metric names, types, labels, or help text remain on the page.
