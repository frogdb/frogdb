# Spec: operations/configuration.md

Status: rewrite
Audiences: A4 (primary), A1, A5

Goal: The reader understands FrogDB's four configuration surfaces — the TOML
file, `frogdb-server` CLI flags, `FROGDB_`-prefixed environment variables, and
runtime `CONFIG GET`/`CONFIG SET` — knows the exact precedence between them,
and can tell which parameters are runtime-mutable, which are startup-only, and
which are accepted-but-ignored Redis compatibility no-ops. This page is the
conceptual *guide* to configuration; the exhaustive parameter tables live in the
Reference section and are linked, not duplicated here.

Not in scope:
- The full parameter table (type/default/section for every field) — that is
  Reference → Configuration reference (`reference/configuration`, which owns the
  tables generated from `config-reference.json`).
- The annotated example config file — that is Reference → Example config
  (generated by `--generate-config`, work-item S4).
- Per-feature semantics (durability modes, TLS modes, ACL rules) — those live on
  their own Operations pages; link to them.
- Any hand-copied list of individual parameters. If the page needs to show
  parameters, it embeds the generated component, it does not retype them.

Sources of truth (author must read before writing):
- `frogdb-server/crates/server/src/config/loader.rs` — `Config::load` (lines
  ~66–207) is the single assembly point. It uses the **figment** crate. This
  file defines the entire precedence chain and the env-var mapping. READ IT.
- `frogdb-server/crates/config/src/lib.rs` — root `Config` struct (~67–185),
  `#[serde(deny_unknown_fields, rename_all = "kebab-case")]`. Authoritative list
  of top-level TOML section names.
- `frogdb-server/crates/server/src/main.rs` — clap CLI definition (~18–109) and
  `--generate-config` handling (~114–119).
- `frogdb-server/crates/config/src/params.rs` — `config_param_registry()` and
  the `ConfigParamInfo` struct. The single source of truth for CONFIG GET/SET
  names, mutability, and no-op status.
- `frogdb-server/crates/server/src/runtime_config.rs` — how CONFIG GET/SET
  consult the registry (glob filtering, `ImmutableParameter` error, no-op
  handling under `strict_config`).

## Verified facts (use these; do not re-derive from prose docs)

**Four surfaces, one assembly point.** Everything is merged by `Config::load`
in `loader.rs` on top of figment.

**Precedence, lowest → highest wins:**

1. Built-in defaults (`Serialized::defaults(Config::default())`)
2. TOML file (`--config PATH`, else `frogdb.toml` in cwd; missing implicit file
   is a warning + defaults, missing explicit `--config` is an error)
3. Environment variables (`FROGDB_`-prefixed, merged via figment)
4. CLI flags (applied imperatively *after* the figment extract, so they always
   win)

`CONFIG SET` is a **separate axis** applied to the live process after startup —
it is not part of the load-time merge. State this explicitly so readers do not
assume `CONFIG SET` edits the file (it does not; it mutates in-memory state, and
`ACL SAVE` / a file rewrite is the only persistence path where one exists).

**Environment-variable mapping** (`loader.rs` ~96–105): prefix `FROGDB_`,
section separator is a **double underscore** `__`, and single underscores within
a name become hyphens (to match kebab-case serde renames). So:
- `FROGDB_SERVER__BIND=0.0.0.0` → `server.bind`
- `FROGDB_MEMORY__MAXMEMORY_POLICY=allkeys-lru` → `memory.maxmemory-policy`
- `FROGDB_LATENCY_BANDS__ENABLED=true` → `latency-bands.enabled`
(`RUST_LOG` and `FROGDB_FLAME_OUTPUT` are read directly, outside the figment
merge — mention only if relevant.)

**CLI flags are a subset**, not a mirror of every field. The real flags:
`--config/-c`, `--bind/-b`, `--port/-p`, `--shards/-s`, `--log-level/-l`,
`--log-format`, `--admin-bind`, `--admin-port`, `--http-bind`, `--http-port`,
`--http-token`, `--tls-enabled`, `--tls-cert-file`, `--tls-key-file`,
`--tls-ca-file`, `--tls-port`, `--tls-require-client-cert`, `--tls-replication`,
`--tls-cluster`, `--generate-config`, `--intrinsic-latency`,
`--startup-latency-check`. Do not hand-list these on this page — link to
Reference → frogdb-server CLI (S5). Describe the *pattern* (flags override a
subset; TLS flags bundle into `TlsCliOverrides`).

**Registry counts (VERIFY-BEFORE-WRITING — the task brief said 61; source says
60):** `config_param_registry()` has **60** entries, broken down as:
- 31 mutable and real (take effect via `CONFIG SET`) — e.g. `maxmemory`,
  `maxmemory-policy`, `loglevel`, `durability-mode`, `sync-interval-ms`,
  `requirepass`, `maxclients`, `lua-time-limit`.
- 14 mutable but **no-op** (accepted for Redis-client compatibility, ignored
  internally; `section`/`field` are `None`) — e.g. `save`, `hz`, `activedefrag`,
  `list-max-listpack-size`, `zset-max-listpack-entries`.
- 15 **read-only** (startup-only; `CONFIG SET` returns `ImmutableParameter`) —
  e.g. `bind`, `port`, `num-shards`, `dir`, `persistence-enabled`, and 8
  `tls-*` params.
The page must NOT hardcode any of these counts in prose (content policy §6). If
a count is shown, it comes from the generated data or is omitted. Re-count from
source at write time; do not trust "60" or "61" as a literal to type.

**`--generate-config` (S4) prints to stdout and exits — it does NOT write a
file** despite its clap help text. Show `frogdb-server --generate-config >
frogdb.toml`. IMPORTANT caveat the page must state honestly: the emitted
template is a **hand-maintained string** (`default_toml_impl()` in `loader.rs`),
separate from `Config::default()`, and it **omits several valid sections**
(`blocking`, `hotshards`, `debug-bundle`, `compat`, `tiered-storage`,
`monitor`). So "the example config is not exhaustive; the Configuration
reference is the complete list." (This is also a drift risk flagged below.)

**`deny_unknown_fields`** is set on the root config — an unknown TOML key aborts
startup. Worth a one-line "typos fail fast" note.

Structure (H2/H3 outline — one line each):

- Intro (2–3 sentences): FrogDB reads configuration from four surfaces; this
  page explains each and how they combine. Link to Reference → Configuration
  reference for the exhaustive parameter list.
- **## The four surfaces** — a short table: surface / how you set it / when it
  applies (startup vs runtime) / typical use. TOML file, CLI flags, env vars,
  `CONFIG SET`.
- **## Precedence** — the ordered list (defaults < TOML < env < CLI), stated
  plainly, with the one subtlety that CLI flags are stamped on last and always
  win, and that `CONFIG SET` is a separate runtime axis that does not write back
  to the file. A tiny worked example (same key set three ways → which wins).
- **## TOML configuration file** — how the file is located (`--config` vs
  implicit `frogdb.toml`), that sections are kebab-case, that unknown keys are
  rejected (`deny_unknown_fields`). Link to the generated example config and the
  reference table. One short real snippet (2–3 sections) only.
- **## Command-line flags** — flags override a subset of fields at startup;
  point to Reference → frogdb-server CLI for the full list. Note
  `--generate-config` here with the stdout-redirect example and the
  "not-exhaustive template" caveat.
- **## Environment variables** — the `FROGDB_` prefix, `__` section separator,
  `_`→`-` rule, with the three worked examples above. Note this suits container
  deployments (tie to Deployment page).
- **## Runtime configuration (CONFIG GET / CONFIG SET)** — what is mutable at
  runtime vs startup-only vs no-op. Explain the three categories (real-mutable,
  Redis-compat no-op, read-only) and that `CONFIG SET` on a read-only param
  returns an error, and that no-op params are accepted for tool compatibility
  (and hidden under `strict_config`). Embed/reference the generated mutability
  column rather than listing params. Note `CONFIG SET` does not persist to the
  TOML file.
- **## See also** — links: Reference → Configuration reference, Reference →
  Example config, Reference → frogdb-server CLI, and the feature pages
  (Persistence, Security/TLS, Observability) for the meaning of specific params.

Generated data:
- `config-reference.json` (via `docs-gen`, `just docs-gen`) — carries per-field
  `type`/`default`/`description` and, for params that map to a registry entry, a
  `mutable` flag. Produced by `frogdb-server/ops/docs-gen/src/main.rs`, which
  `use`s `config_param_registry()` and joins on `(section, field)`. The
  runtime-mutability column on this page and in the reference table comes from
  here.
- Example config page (S4) — output of `frogdb-server --generate-config`.
- CLI reference (S5) — clap flags for `frogdb-server`; this page links to it
  instead of listing flags.

Drift guards:
- `just docs-gen-check` (`docs-gen -- --check`) — CI fails if
  `config-reference.json` is stale vs the Rust config structs + params registry.
  This keeps the mutability data honest.
- `params.rs` has in-crate tests enforcing registry invariants: no duplicate
  names, every no-op is mutable, no-ops have no config path. `runtime_config.rs`
  has a test cross-checking the live runtime param list against
  `config_param_registry()`. These keep the registry and runtime behavior in
  sync; the page can rely on the registry being authoritative.
- **Known drift risk to call out for future maintainers:** the
  `--generate-config` template (`default_toml_impl()`) is hand-maintained and
  separate from `Config::default()`; there is no `--check` tying it to the real
  section list, so it can silently omit sections (it currently omits `blocking`,
  `hotshards`, `debug-bundle`, `compat`, `tiered-storage`, `monitor`). The page
  must not present the example config as exhaustive, and S4 should ideally add a
  check. Flag this to the doc reviewer.
- No hardcoded parameter counts, version numbers, or crate counts in prose (§6).
