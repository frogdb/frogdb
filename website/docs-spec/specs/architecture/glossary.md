# Spec: architecture/glossary.md
Status: update (error-code table has fabricated/absent codes; several terms are
speculative or now-inaccurate and must be pruned or corrected)
Audiences: A2 (skeptics), A3 (architecture-curious), A5 (contributors)

Goal: The reader has a single, accurate reference for FrogDB terminology, the wire
error codes the server actually emits, and abbreviations used across the docs.
Every term earns its place by appearing in at least one other page; every error
code and message matches source exactly. The reader can resolve any unfamiliar
term encountered elsewhere in the docs from here.

Not in scope:
- Teaching the concepts — the glossary defines and links; the owning pages
  explain. Keep each entry to a definition plus a link to its home page.
- Per-command semantics — the command matrix and redis.io.

Existing content: current `architecture/glossary.md`. Keep the well-formed
slot-vs-shard, hash-tag, WAL, snapshot, epoch, transaction, and abbreviation
sections. FACTUAL DISCREPANCIES and PRUNES to apply:

ERROR-CODE TABLE — rebuild against source. Corrections:
- `-CROSSSLOT`, `-MOVED slot host:port`, `-ASK slot host:port`, `-WRONGTYPE`,
  `-OOM`, `-NOSCRIPT`, `-BUSY`, `-READONLY` — all real; verify the exact message
  text against source (e.g. OOM is "OOM command not allowed when used memory >
  'maxmemory'"; CROSSSLOT is "Keys in request don't hash to the same slot").
- `-TIMEOUT` ("VLL queue … waiting for locks") — **DELETE**. No `-TIMEOUT` code and
  no VLL lock-timeout error exists. Timeouts are emitted with the generic `ERR`
  prefix (`ERR timeout`, `ERR scan timeout`, …). If the page documents timeout
  behavior, do so as `ERR timeout`, not a `-TIMEOUT` code.
- `-NOSCRIPT`/`-BUSY` scripting note is fine; verify BUSY has two forms (script-
  running vs FUNCTION LOAD).
- **ADD** the real codes the table omits: `-CLUSTERDOWN` ("The cluster is down"),
  `-TRYAGAIN` ("Multiple keys request during rehashing of slot"), `-NOAUTH`
  ("Authentication required."), `-NOPERM` (ACL permission messages), `-NOPROTO`,
  `-BUSYKEY`, `-BUSYGROUP`, `-NOGROUP`, `-EXECABORT`, `-VERSIONMISMATCH`.
- **DO NOT** document `-MASTERDOWN`, `-NOREPLICAS`, `-LOADING` as emitted — they
  exist only as passthrough labels/flags and are never produced by FrogDB (FrogDB
  has no RDB loading phase). `-MISCONF` and `-NOREPL` do not exist at all — never
  document them. If replication/consistency prose elsewhere references any of these,
  fix those pages (the replication spec already flags `-NOREPL`).
- `-ASKING` is not an error — it is a command/connection-state keyword; do not list
  it as an error code.

TERM PRUNES / CORRECTIONS:
- **Orchestrator** — DELETE or REDEFINE. The clustering control plane is internal
  Raft consensus (`openraft`), not an external orchestrator; the term as currently
  written ("external control plane … like DragonflyDB … rather than gossip") is
  wrong (see clustering spec). If any page still needs the word, it applies only to
  the Kubernetes operator (a deployment concern). Remove the term from the
  architecture glossary unless a page genuinely uses it in the operator sense, and
  never as the topology control plane.
- **Dashtable** — the entry describes a DragonflyDB structure FrogDB does NOT use
  ("FrogDB uses griddle currently; Dashtable is a future consideration"). Per PLAN
  §6 (no aspirational features), DELETE unless "Dashtable" is actually referenced
  on another page for a real comparative reason; a future-consideration term that
  appears only here fails the "every term used elsewhere" rule.
- **Internal-shard routing formula** — the glossary's slot/shard table says
  Internal Shard uses "xxhash64". This is wrong (it is `slot % num_shards`,
  CRC16-derived); fix the algorithm column to match the clustering/storage fix.
- Verify **Griddle**, **COW**, **Eviction Pool**, **Tail Latency**, **Throughput**
  each appear on at least one other page (storage, persistence, concurrency). Prune
  any that do not, or ensure the owning page uses them.

PRUNE RULE (enforce): every term and abbreviation MUST appear in at least one other
docs page. Before publishing, grep each glossary headword across
`website/src/content/docs/**` (excluding the glossary itself). Remove any term with
zero external uses. Conversely, terms used heavily elsewhere but missing here
(e.g. `RouteDecision`, `config epoch`, `version gate`, `VLL`, `Raft`, `PSYNC`,
`replication ID`) should be considered for inclusion. The glossary must be the
intersection-plus of what the docs actually use — not a superset of invented terms.

Drift guards:
- ERROR-TABLE GUARD: the strongest drift protection is to generate the error table
  from `CommandError` (`define_command_errors!`) rather than hand-maintaining it.
  Proposed work item **S9** (proposed — not yet in PLAN §6; flag for the docs-gen
  owner): dump `CommandError` variants + messages to JSON at build and render the
  table from it, with a `--check` mode. Until then, a reviewer diffs the table
  against `types/src/error.rs` (+ the acl/scripting error modules) on every edit.
- TERM-USAGE GUARD: a build/CI check that every glossary headword appears in at
  least one other page (and, ideally, that links resolve). Flag as S8-adjacent.
  This directly enforces the prune rule and prevents dead terms (Dashtable,
  Orchestrator) from reaccumulating.
- S7 code-path check covers `types/src/error.rs`, `acl/src/error.rs`,
  `core/src/scripting/error.rs`, `connection/guards.rs`.
- No hardcoded counts (slot count 16384 excepted as a protocol invariant); no
  performance numbers in the metrics-terms section.
