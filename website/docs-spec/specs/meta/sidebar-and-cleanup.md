## 3. Reconcile Operations `sidebar.order` frontmatter (collision fix)

Current `sidebar.order` values (verified in source): deployment 1, persistence 2,
replication 3, clustering 4, monitoring 5, observability **6**, backup-restore
**6**, security **7**, debug-ui **7**, performance 8, kubernetes 11. Two
collisions: observability/backup-restore (both 6) and security/debug-ui (both 7).

Important context: the Operations topic uses an **explicit `items` array** in
`astro.config.mjs`, so that array — not frontmatter `sidebar.order` — actually
controls sidebar order for this topic. The frontmatter orders are therefore
**vestigial** and are the reason two collisions went unnoticed. Choose ONE of:

- **(Preferred) Remove `sidebar.order` from all Operations pages.** Single source
  of truth = the config `items` array. This eliminates the collisions permanently
  and prevents future drift between two ordering mechanisms. (Deleted pages —
  monitoring, performance, debug-ui, kubernetes — take their frontmatter with them.)
- **(Alternative) Keep and renumber** to a gap-free sequence matching the new §1
  Operations order: configuration 1, deployment 2, persistence 3, replication 4,
  clustering 5, security 6, observability 7, diagnostics 8, backup-restore 9.

Whichever is chosen, the end state must have no duplicate order among the surviving
Operations pages, and the frontmatter (if kept) must agree with the config array
order. Do NOT leave the current 6/6 and 7/7 collisions.

Note: Architecture uses `autogenerate`, so its frontmatter `sidebar.order` DOES
matter and must remain internally consistent (replication 7, clustering 8,
consistency 9, protocol 12, connection 13, debugging 15, glossary 16 today —
verify no collisions there after `testing.md` leaves; testing.md's order slot
frees up but does not need backfilling).

## 4. CI / build interaction (PLAN §6 S8)

- Adding/removing pages and redirects must not break `just docs-build`. Confirm the
  `starlightLinksValidator` passes after the moves (it runs in the Starlight
  plugin chain). Broken internal links fail the build — treat that as the
  acceptance gate.
- This structural change is a prerequisite for several page specs (Diagnostics,
  Configuration, Example config, Testing methodology, Extensions). Sequence: create
  the new/moved pages first (or as stubs), then flip the topic array and delete old
  files, so the site never references a nonexistent slug mid-change.

Drift guards:
- The topic array in `astro.config.mjs` is the single nav source of truth; the
  `sidebar.order` reconciliation removes the competing mechanism for Operations.
- Every deleted slug has a redirect; a periodic check (or the links validator plus
  a redirect-coverage review) ensures no dangling references. Flag a small
  follow-up (S8-adjacent) to assert every path in the `redirects` map resolves to a
  live page.
- S7 code-path check is unaffected (it validates `crates/...` paths in prose, not
  nav), but the moved Testing methodology page must keep its `crates/testing` /
  `testing/jepsen` references valid under S7 at its new location.
