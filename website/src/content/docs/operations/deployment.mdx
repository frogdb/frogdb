This guide covers deployment patterns and best practices for running FrogDB in production.

## Single Node Deployment

The simplest deployment for applications that don't require high availability:

```toml
# production.toml
[server]
bind = "0.0.0.0"
port = 6379
max_clients = 10000

[persistence]
enabled = true
data_dir = "/var/lib/frogdb"
mode = "hybrid"
aof_fsync = "everysec"

[memory]
maxmemory = "4gb"
maxmemory_policy = "allkeys-lru"

[logging]
level = "info"
format = "json"
```

### Systemd Service

```ini
# /etc/systemd/system/frogdb.service
[Unit]
Description=FrogDB Server
After=network.target

[Service]
Type=simple
User=frogdb
Group=frogdb
ExecStart=/usr/local/bin/frogdb-server --config /etc/frogdb/frogdb.toml
Restart=always
RestartSec=5
LimitNOFILE=65535

# Security hardening
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
ReadWritePaths=/var/lib/frogdb

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable frogdb
sudo systemctl start frogdb
```

## High Availability Setup

### Primary with Replicas

Deploy one primary and multiple replicas:

```
                    ┌────────────┐
      writes ──────▶│  Primary   │
                    └─────┬──────┘
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
      ┌──────────┐  ┌──────────┐  ┌──────────┐
      │ Replica 1│  │ Replica 2│  │ Replica 3│
      └──────────┘  └──────────┘  └──────────┘
            │             │             │
            └─────────────┴─────────────┘
                          │
                        reads
```

**Primary configuration:**

```toml
# primary.toml
[server]
bind = "0.0.0.0"
port = 6379

[persistence]
enabled = true
data_dir = "/var/lib/frogdb"
mode = "hybrid"
```

**Replica configuration:**

```toml
# replica.toml
[server]
bind = "0.0.0.0"
port = 6379

[replication]
role = "replica"
primary_host = "primary.internal"
primary_port = 6379

[persistence]
enabled = true
data_dir = "/var/lib/frogdb"
```

### Load Balancer Setup

Use HAProxy or similar for read distribution:

```haproxy
# haproxy.cfg
frontend frogdb_write
    bind *:6379
    default_backend frogdb_primary

frontend frogdb_read
    bind *:6380
    default_backend frogdb_replicas

backend frogdb_primary
    server primary 192.168.1.10:6379 check

backend frogdb_replicas
    balance roundrobin
    server replica1 192.168.1.11:6379 check
    server replica2 192.168.1.12:6379 check
    server replica3 192.168.1.13:6379 check
```

## Docker Deployment

### Docker Compose

```yaml
# docker-compose.yml
version: '3.8'

services:
  frogdb-primary:
    image: frogdb/frogdb:latest
    ports:
      - "6379:6379"
    volumes:
      - frogdb-data:/data
      - ./frogdb.toml:/etc/frogdb/frogdb.toml:ro
    command: ["frogdb-server", "--config", "/etc/frogdb/frogdb.toml"]
    deploy:
      resources:
        limits:
          memory: 4G
    healthcheck:
      test: ["CMD", "frogdb-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  frogdb-replica:
    image: frogdb/frogdb:latest
    depends_on:
      frogdb-primary:
        condition: service_healthy
    volumes:
      - frogdb-replica-data:/data
    environment:
      - FROGDB_REPLICATION_ROLE=replica
      - FROGDB_REPLICATION_PRIMARY_HOST=frogdb-primary
      - FROGDB_REPLICATION_PRIMARY_PORT=6379
    deploy:
      replicas: 2
      resources:
        limits:
          memory: 4G

volumes:
  frogdb-data:
  frogdb-replica-data:
```

## Kubernetes Deployment

### StatefulSet

```yaml
# frogdb-statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: frogdb
spec:
  serviceName: frogdb
  replicas: 3
  selector:
    matchLabels:
      app: frogdb
  template:
    metadata:
      labels:
        app: frogdb
    spec:
      containers:
      - name: frogdb
        image: frogdb/frogdb:latest
        ports:
        - containerPort: 6379
          name: frogdb
        - containerPort: 16379
          name: cluster
        volumeMounts:
        - name: data
          mountPath: /data
        resources:
          requests:
            memory: "2Gi"
            cpu: "500m"
          limits:
            memory: "4Gi"
            cpu: "2"
        livenessProbe:
          exec:
            command: ["frogdb-cli", "ping"]
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          exec:
            command: ["frogdb-cli", "ping"]
          initialDelaySeconds: 5
          periodSeconds: 5
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
---
apiVersion: v1
kind: Service
metadata:
  name: frogdb
spec:
  clusterIP: None
  ports:
  - port: 6379
    name: frogdb
  selector:
    app: frogdb
```

## Resource Planning

### Memory Sizing

FrogDB stores data primarily in memory. Plan for:

- **Data size** — Your working dataset
- **Overhead** — ~20% for internal structures
- **Buffer** — 10-20% headroom for operations

```
Required Memory = (Data Size × 1.2) + Buffer
```

Example: 5GB of data → plan for 7-8GB memory

### CPU Sizing

FrogDB is single-threaded for command processing but uses multiple threads for:
- Network I/O
- Background persistence
- Replication

Recommendations:
- **Light workload**: 1-2 cores
- **Medium workload**: 2-4 cores
- **Heavy workload**: 4+ cores

### Disk Sizing

For persistent deployments:

```
Disk = Data Size × 2 (for snapshots) + AOF growth buffer
```

Use fast SSDs for best persistence performance.

## Security

### Network Security

- Bind to private interfaces only
- Use firewalls to restrict access
- Consider VPN for cross-datacenter replication

```toml
[server]
bind = "10.0.0.10"  # Private interface only
```

### TLS Encryption

```toml
[server]
tls_cert = "/etc/frogdb/tls/server.crt"
tls_key = "/etc/frogdb/tls/server.key"
```

### Authentication

```toml
[auth]
requirepass = "your-secure-password"
```

Connect with password:

```bash
redis-cli -a your-secure-password
```

## Operational Checklist

Before going to production:

- [ ] Configure appropriate `maxmemory` and eviction policy
- [ ] Enable persistence with appropriate durability settings
- [ ] Set up monitoring and alerting
- [ ] Configure log rotation
- [ ] Test backup and restore procedures
- [ ] Document failover procedures
- [ ] Set appropriate OS limits (`ulimit -n`)
- [ ] Disable THP (Transparent Huge Pages)
- [ ] Configure proper permissions on data directory
