# Area B: Transactions / Scripting / Cross-shard (agent report, verified evidence)

### 1. `subscribe-family-in-multi-diverges-and-untested`
**Pub/sub subscription commands inside MULTI silently diverge from Redis and are untested either way.** `queue_command` (`frogdb-server/crates/server/src/connection/guards.rs:470-544`) has no NO_MULTI rejection — it happily queues `SUBSCRIBE/PSUBSCRIBE/SSUBSCRIBE` (it even runs their channel-ACL check at line 524). At EXEC, `exec_pubsub_in_transaction` (`frogdb-server/crates/server/src/connection/pubsub_conn_command.rs:958-978`) *actually subscribes* for `SUBSCRIBE/UNSUBSCRIBE/PSUBSCRIBE/PUNSUBSCRIBE`, but rejects `SSUBSCRIBE/SUNSUBSCRIBE/PUBSUB` with a bespoke `"ERR command not supported inside MULTI"`. Grep of `multi_regression.rs`/`multi_tcl.rs`/`integration_transactions.rs` finds zero subscribe-in-MULTI tests. RESET inside MULTI (intercepted at `dispatch.rs:373`, `ResetIntercept`) is likewise untested (`needs:reset` excluded from `multi_tcl.rs:5`).
- **expected (Redis 8.6):** `SUBSCRIBE/UNSUBSCRIBE/PSUBSCRIBE/PUNSUBSCRIBE/SSUBSCRIBE/SUNSUBSCRIBE/RESET` carry `CMD_NO_MULTI`: queued inside MULTI they return `-ERR <cmd> is not allowed in transactions` immediately and mark the tx dirty → EXEC returns `EXECABORT`. FrogDB instead (a) executes the sharded/plain subscribe families, and (b) uses two different, non-Redis error strings/timings for the shard family. Least-surprise: at minimum the whole subscribe family should behave uniformly; the current split (plain works, sharded errors at EXEC) is a self-inconsistency bug regardless of Redis parity.
- **suggested tests:** integration — `MULTI; SUBSCRIBE ch; EXEC` and the same for each of the six subscribe variants + RESET; assert the chosen policy (recommend: queue-time `-ERR ... not allowed in transactions` + `EXECABORT`, matching Redis and removing the EXEC-time special-casing). Add a RESET-during-MULTI test asserting the queue is dropped and `+RESET` returned.
- **likelihood 3** — confirmed live divergence + internal inconsistency, wholly untested. **consequence 2** — wrong results (an app subscribing inside MULTI ends up subscribed on FrogDB but errored on Redis; the sharded/non-sharded split is a latent bug).

### 2. `allow-cross-slot-standalone-must-not-relax-multi` (invariant unguarded)
**No test pins that `allow_cross_slot_standalone=true` does NOT extend cross-shard support to MULTI/EXEC.** `fold_transaction_keys` (`frogdb-server/crates/server/src/connection/state.rs:859-868`) and `TransactionTarget::resolve` (`state.rs:42-47`) unconditionally map a shard-spanning transaction to `Multi → Err(crossslot())` — they never consult the config. This is the core arch guarantee: VLL gives cross-shard *multi-key ops* execution atomicity, but that support is deliberately withheld from transactions. The only standalone cross-shard-MULTI CROSSSLOT test is the WATCH-fold control case (`integration_transactions.rs:360-380`); there is no test with `allow_cross_slot_standalone=true` proving a MULTI over keys on different shards *still* CROSSSLOTs (grep: `allow_cross_slot` appears in `integration_transactions.rs` zero times).
- **expected (least-surprise, arch doc):** cross-shard MULTI → `CROSSSLOT` even when standalone cross-slot single ops are enabled. A future refactor wiring the config into `fold` would silently permit a non-atomic, non-rollback cross-shard transaction — a guarantee violation with no failing test.
- **suggested tests:** integration — start standalone (4 shards) with `allow_cross_slot_standalone: true`; `MULTI; SET {a}… ; SET {b}…` (keys on different shards); assert EXEC returns `CROSSSLOT`. Add the mirror with `false`. Also a plain-key (no WATCH) cross-shard MULTI CROSSSLOT test for the default config, which is currently missing.
- **likelihood 2** — the invariant is load-bearing and the config is an obvious future lever. **consequence 3** — if broken, cross-shard EXEC runs with VLL execution-atomicity but no failure-atomicity/rollback → partial commits / data-integrity violation.

### 3. `cross-slot-multi-weak-assertion`
**The only cluster cross-slot MULTI test asserts almost nothing.** `test_multi_exec_cross_slot_returns_error` (`frogdb-server/crates/server/tests/integration_cluster.rs:7846-7884`) accepts an error on *any* of MULTI, either queued SET, or EXEC, and its own comment says "The exact behavior depends on implementation." FrogDB actually aborts at queue time: `try_queue_in_transaction` → `validate_cluster_slots` → `abort_transaction` (`guards.rs:454-461`), so the second (different-slot) SET should error at queue time and EXEC should return `EXECABORT`. The test pins none of this, so a regression flipping queue-time-abort to a silent EXEC-time partial would still pass.
- **expected:** queue-time — the cross-slot queued command returns a `CROSSSLOT` error and marks the tx aborted; EXEC returns `EXECABORT`. (Confirm the exact FrogDB string and lock it.)
- **suggested tests:** tighten to assert the *specific* command that errors (the second SET), its `CROSSSLOT` prefix, and that EXEC returns `EXECABORT` — not "errors somewhere."
- **likelihood 2** — weak assertion masks the real contract. **consequence 1** — compat/regression-detection nit (behavior is currently correct; the test just can't catch a break).

### 4. `wait-in-multi-untested`
**WAIT inside MULTI has a claimed contract with zero tests.** `WaitCommand::execute` (`frogdb-server/crates/server/src/commands/replication.rs:560-576`) has a "Deny-blocking context (MULTI/EXEC …)" branch returning `count_acked(current_offset)` without blocking. But WAIT's real blocking path is the connection-layer `WaitCoordinator`; whether a *queued* WAIT reaches this non-blocking shard branch (vs. the blocking coordinator) at EXEC time is unverified. `wait_tcl.rs` tests only standalone arg-validation and no-replica cases; grep finds no `WAIT`-in-MULTI test anywhere.
- **expected (Redis 8.6):** WAIT is queued in MULTI and at EXEC returns the acknowledged-replica count immediately (never blocks — `CLIENT_DENY_BLOCKING`). On standalone/no-replicas it returns `0`.
- **suggested tests:** integration — `MULTI; WAIT 0 0; EXEC` and `MULTI; WAIT 3 100; EXEC` on a no-replica standalone; assert EXEC returns within a tight timeout with `Integer(0)` at the WAIT slot (guards against a hang).
- **likelihood 2** — plausible the queued path blocks or mis-counts; entirely unverified. **consequence 2** — a blocking regression hangs EXEC (availability); a mis-count returns wrong results.

### 5. `blmpop-bzmpop-nonblocking-in-multi-omitted`
**Two blocking commands are missing from the non-blocking-in-MULTI regression.** `tcl_blocking_commands_ignore_timeout_in_multi` (`frogdb-server/crates/redis-regression/tests/multi_tcl.rs:516-578`) queues and EXEC-verifies nil for BLPOP/BRPOP/BRPOPLPUSH/BLMOVE/BZPOPMIN/BZPOPMAX/XREAD BLOCK/XREADGROUP — but omits `BLMPOP` and `BZMPOP`, both of which exist and are blocking (`frogdb-server/crates/commands/src/blocking.rs`; registered in `register.rs`). The Lua sibling (`scripting_tcl.rs:462-576`, `tcl_eval_scripts_do_not_block_on_*`) similarly omits BLMPOP/BZMPOP/XREAD BLOCK.
- **expected (Redis 8.6):** inside MULTI (and inside Lua) BLMPOP/BZMPOP run non-blocking with timeout ignored, returning a null array on empty keys.
- **suggested tests:** extend the MULTI test with `BLMPOP 0 1 empty LEFT` and `BZMPOP 0 1 emptyz MIN`, asserting null-array at EXEC; add BLMPOP/BZMPOP/`XREAD BLOCK` to the Lua non-block test set.
- **likelihood 1** — these likely share the generic deny-blocking path already exercised by siblings. **consequence 2** — if either slipped the path, EXEC (or a script) hangs forever on an empty key.

### 6. `dead-undeclared-key-validation-and-misdocumented-behavior`
**The "strict undeclared-key" deviation is dead code; the real enforcement is CROSSSLOT-based, and neither the allow nor the reject case is pinned.** `validate_key_access` and `ScriptError::UndeclaredKey` (`frogdb-server/crates/core/src/scripting/bindings.rs:82-92`, `error.rs:22-23`) are `#[allow(dead_code)]` with only isolated unit tests — grep shows no production caller. Actual script key safety is the CROSSSLOT tracker: `ScriptCommandGate::classify` → `cross_slot.check_all` (`frogdb-server/crates/core/src/scripting/gate.rs:213-215`), seeded from the first declared key's slot (`executor.rs:316-324`). So a cluster/shebang script accessing an *undeclared key in the same slot* is **allowed**, and one accessing a *different slot* gets `CROSSSLOT` — not the "errors immediately on undeclared keys" the `scripting_tcl.rs:19-25` exclusion header claims. No test pins the same-slot-undeclared = allowed case, nor the numkeys=0 cluster case (seed `None`, first accessed key sets the slot — `executor.rs:320-324`).
- **expected:** document + test the true policy: undeclared key access is permitted as long as it stays within the seeded slot; cross-slot access → `CROSSSLOT`. Remove or wire the dead `validate_key_access` (leaving it invites a future dev to enable it and break standalone numkeys=0 scripts, which the existing `test_eval_undeclared_key_access_standalone` at `integration_scripting.rs:342` expects to *succeed*).
- **suggested tests:** cluster/shebang — script with `numkeys=1` declaring `{tag}a` that `redis.call('get', '{tag}b')` (same slot, undeclared) → success; same script touching a different slot → `CROSSSLOT`; a `numkeys=0` shebang script touching two different-slot keys → `CROSSSLOT`. Add a `dead_code`/coverage guard or delete `validate_key_access`.
- **likelihood 2** — dead code + misdocumented contract; the untested same-slot-undeclared path is exactly where a "fix" would regress. **consequence 2** — wrong results / spurious errors for scripts; a mistaken activation breaks standalone scripting compatibility.
